## 1. 适用范围

本文适用于通过 Windows 本地服务、Python 和 `lark-cli.cmd` 向飞书发送角色回复的项目，尤其适合以下情况：

- 模型已经生成完整长回复，但飞书只显示第一段；
- 本地数据库显示 300 字以上，用户实际只收到约 100 字；
- 发送接口返回了 `message_id`，系统因此误以为全文已经送达；
- 中文、换行或标点经过 Windows 命令行后发生截断或乱码；
- 多个角色共用相似的飞书桥接代码，需要统一可靠性规范。

核心结论：**生成成功、命令执行成功、取得消息回执和用户收到完整正文，是四件不同的事。** 不要用前一种证据代替后一种证据。

## 2. 本次故障的本质

原发送方式把模型生成的多段正文作为 `--text` 后面的原始命令行参数，直接传给 Windows 的 `.cmd` 入口：

```python
[
    "lark-cli.cmd", "im", "+messages-send",
    "--chat-id", chat_id,
    "--text", text,
]
```

这种写法在普通单行文本上可能长期正常，但多段正文包含真实换行时，`.cmd`、命令行参数解析和当前控制台代码页共同构成了不可靠边界。典型结果是：

1. 模型生成了三段完整正文；
2. 本地 outbox 保存了三段；
3. 命令返回成功并给出飞书 `message_id`；
4. 飞书服务端实际只保存第一段；
5. 系统把本地三段全部写进角色正史，形成“角色以为用户看过，用户实际上没看过”的证据错位。

这不是模型长度问题，也不是飞书不支持长消息，而是**发送载荷在渠道边界被破坏**。

## 3. 可靠修复：ASCII-safe JSON 单帧传输

不要让原始中文和真实换行直接穿过 `.cmd` 参数。先把完整正文编码成单行、纯 ASCII 的 JSON，再使用 `--content` 发送：

```python
import json

content = json.dumps(
    {"text": text},
    ensure_ascii=True,
    separators=(",", ":"),
)

args = [
    "im", "+messages-send",
    "--chat-id", chat_id,
    "--msg-type", "text",
    "--content", content,
    "--idempotency-key", idempotency_key,
    "--as", "bot",
]
```

这里三个选择缺一不可：

- `--content`：发送明确的 JSON 载荷，而不是把正文作为裸文本参数；
- `ensure_ascii=True`：将中文转成 `\uXXXX`，避开 Windows `.cmd` 的代码页损坏；
- 单行 JSON：将真实换行编码成 `\n`，避开换行造成的参数截断。

飞书 CLI 解析 JSON 后，会还原原始中文和换行。角色看到的仍是正常中文多段消息，ASCII 转义只存在于内部传输边界。

### 为什么不能只用 `ensure_ascii=False`

`ensure_ascii=False` 虽然能把换行转义成 `\n`，却仍会把中文原样放进 `.cmd` 参数。在某些 Windows 环境中，中文可能在当前代码页下变成问号。因此“JSON 化”只能解决换行问题，**ASCII 化才同时解决换行和中文编码问题**。

## 4. 与文风和字数预算的关系

发送完整性不能代替表达设计，但表达设计必须经过可靠发送边界。新角色项目应把“说什么、说多长”和“怎样完整送达”分成两个契约。

### 4.1 文风是角色专属配置

不要把某个已成功角色的称呼、撒娇方式、亲密程度、玩笑类型或固定尾巴复制给所有新角色。可复用的是组织原则：

- 每轮围绕一个中心反应，不把回复拼成互不相干的段落；
- 回应用户当轮字面信息和合理潜台词；
- 给出角色自己的具体态度、观察或内容；
- 提供能让互动继续生长的具体内容，但不强制用问题收尾；
- 每轮最多使用一套主要玩笑机制；
- 禁止同义复述、动作灌水、连续比喻和通用亲密尾巴凑字；
- 严肃语境优先解决问题，轻松、暧昧或成人玩笑只在角色设定、成年边界和当轮语境都允许时出现。

角色称呼、句式、攻击性、柔软度、叙述视角、成人玩笑尺度和是否常用尾巴，都应写进该角色自己的版本化表达档案。

### 4.2 可选的饱满回复预算

当用户明确希望“每次回复看起来够饱”时，可以采用下面这套已验证结构作为起点，而不是所有角色的强制默认值：

- `normal`：目标 300–500 个非空白可见字符，最低 300；
- `closing`：目标 100–200 个非空白可见字符，最低 100；
- 标点是否计数必须在项目中唯一、明确地定义，生成校验、日志和人工验收使用同一口径；
- 问题、新请求、转折或继续话题内容一律归入 `normal`；
- 只有纯确认、告别、睡觉、上班、工作或明确暂停聊天才可归入 `closing`；
- 普通回复至少包含“实际回应、角色自身内容、互动生长点”三个有效内容单元；
- 首稿不足最低值时最多进行两次完整重写，总计不超过三稿；不得在末尾续写凑字；
- 第三稿仍不足时发送第三稿并记录结构化长度警告，不无限重试；
- 只提交最终选中稿的消息、记忆、关系证据和偏好候选，废弃稿不得进入正式状态。

若新角色偏好简洁、克制或即时通讯感，应重新设定预算，不能机械套用 300–500。无论采用哪套预算，最终都必须核对飞书服务端实际正文，而不是只统计本地生成稿。

### 4.3 推荐的职责划分

```text
角色表达档案：文风、语境、内容密度、长度分类与重写策略
模型适配器：生成、计数、最多三稿和最终稿选择
渠道适配器：ASCII-safe JSON 封帧、幂等发送与回执
验收层：本地最终稿、飞书服务端正文和用户显示的一致性
```

不要把字数预算塞进飞书发送器，也不要让渠道适配器自行扩写、截短或改写角色正文。

## 5. 正确的诊断顺序

遇到“回复还是很短”时，不要先改提示词，也不要先换模型。按以下证据链逐层检查：

```text
模型最终稿
→ 本地暂存正文
→ CLI 最终请求载荷
→ 飞书发送回执
→ 根据 message_id 读取的飞书服务端正文
→ 用户客户端实际显示
```

推荐同时记录三种长度，但不要记录敏感正文到普通运行日志：

- `generated_visible_chars`：模型最终稿去空白后的字符数；
- `staged_visible_chars`：写入 outbox 的字符数；
- `provider_visible_chars`：从飞书服务端读回正文后的字符数。

如果前两者相同而第三者明显变短，问题就在渠道发送边界，不在表达系统。

只查看本地 outbox 会产生误判。只拿到 `message_id` 也不能证明正文完整，因为回执只能证明飞书接受了一条消息请求，不能证明请求中的正文与本地最终稿逐字一致。

## 6. 必须具备的测试

### 6.1 单元测试：正文可逆

至少使用包含中文、标点、单换行和空行的三段文本：

```python
message = "第一段。\n\n第二段，保留换行。\n\n第三段。"

content = build_text_content(message)

assert "\n" not in content
content.encode("ascii")
assert json.loads(content) == {"text": message}
```

同时断言发送命令：

- 使用了 `--content`；
- 明确使用 `--msg-type text`；
- 不再出现 `--text`；
- idempotency key 保持稳定；
- JSON 反序列化后与模型最终稿逐字一致。

### 6.2 Windows 真实命令边界探针

Mock 无法覆盖 `.cmd` 的参数解析和代码页。必须在目标 Windows 环境调用真实 `lark-cli.cmd` 的 `--dry-run`：

1. 构造三段中文测试文本；
2. 使用与生产相同的 JSON 编码；
3. 执行 `--dry-run`，不得真实发送；
4. 读取预览请求中的 `body.content`；
5. 连续反序列化必要的 JSON 层；
6. 断言还原后的正文与原文完全相等。

这个探针应明确输出：`dry_run=true`、`unicode_round_trip=true`、段落数量和 `externally_delivered=false`。

### 6.3 飞书真实交付验收

上线前或用户自然发起下一轮聊天后，完成一次服务端核验：

1. 保存本地最终稿的字符数和摘要哈希；
2. 取得飞书返回的 `message_id`；
3. 根据该 ID 从飞书读取实际消息；
4. 提取 `content.text`；
5. 与本地最终稿做逐字比较或规范化后的哈希比较；
6. 确认用户端显示为一条完整多段消息。

行政测试消息不得未经确认主动发送，也不要写入角色正史。优先使用用户自然发起的真实对话完成验收。

## 7. 正史与回执的证据边界

角色系统最危险的并不是“少显示两段”，而是把未送达内容写成双方已经共同经历的事实。

推荐状态流：

```text
generated
→ staged
→ sending
→ receipt_received
→ provider_verified（可选高保证模式）
→ canon_committed
```

最低要求是取得稳定的消息回执后才能提交正史。对于曾发生过正文截断的桥接器，建议增加一段时间的高保证模式：飞书服务端正文核验通过后才提交记忆、关系证据和“用户已知信息”。

如果服务端正文与本地稿不一致：

- 标记为 `delivery_corrupt` 或等价的有限错误类型；
- 不把未送达部分提交为用户已知事实；
- 保留相同 idempotency key 和可审计的发送意图；
- 不自动重复轰炸用户；
- 修复后由明确策略决定补发、放弃或人工对账。

历史上已经发生的错记不要自动重发。先列出“本地记录正文”和“飞书实际正文”的差异，再决定是否需要一次显式修复事件。

## 8. 发布流程

适用于每个角色项目的标准步骤：

1. 下线或 disarm 文字发送能力；
2. 备份数据库并验证完整性；
3. 只修改共享或角色专属的渠道适配器；
4. 补三段中文、空行、标点和长文本回归测试；
5. 运行完整离线测试与编译检查；
6. 在目标 Windows 环境运行真实 `.cmd --dry-run`；
7. 确认 JSON、中文和段落全部可逆；
8. 只重启目标角色的服务链；
9. 核对飞书身份、文字能力、数据库和队列健康；
10. 恢复原有启用状态，其他能力保持原样；
11. 用下一次自然对话做真实服务端正文验收。

切勿因为共用代码而一次性重启所有角色。先在一个角色上通过完整验收，再把同一适配器修复迁移到其他角色。

## 9. 常见错误做法

- 看到回复短就继续堆提示词或强迫模型重写；
- 只统计本地数据库，不读取飞书实际消息；
- 把成功退出码或 `message_id` 当成全文交付证明；
- 用 `--text <多行中文>` 直接穿过 Windows `.cmd`；
- 改成 JSON 后仍使用 `ensure_ascii=False`，忽略代码页；
- 只写 Mock 测试，不经过真实 `.cmd --dry-run`；
- 测试修复时主动发送行政消息，并污染角色对话与正史；
- 发送失败或正文不一致后自动重复发送，造成用户收到多份消息；
- 修复一个角色时顺手重启其他角色服务。

## 10. 可复用验收清单

- [ ] 模型最终稿与 outbox 正文一致
- [ ] 多段正文通过单行 ASCII JSON 传输
- [ ] JSON 反序列化后逐字恢复原文
- [ ] 参数中没有真实换行和非 ASCII 字符
- [ ] 不再使用裸 `--text` 传递多段中文
- [ ] 单元测试覆盖中文、标点、单换行和空行
- [ ] 目标 Windows 环境的 `.cmd --dry-run` 通过
- [ ] 真实飞书消息正文与本地稿一致
- [ ] idempotency key 在重试中保持稳定
- [ ] 未完整送达的内容没有写成用户已知正史
- [ ] 数据库已备份且完整
- [ ] 只重启目标角色服务
- [ ] 其他未授权能力仍保持关闭

## 11. 一句话原则

**长回复是否成功，不看模型写了多少，也不只看本地存了多少；要看经过真实 Windows 命令边界后，飞书服务端实际保存并向用户展示了多少。**
